# Постмортем: сбой GitHub 21-22 октября 2018 года

## Краткое резюме

21 октября 2018 года в 22:52 UTC на GitHub начался крупный инцидент, который привёл к деградации части сервисов на 24 часа 11 минут. Полное восстановление завершилось 22 октября в 23:03 UTC.

Сбой затронул не Git-репозитории, а метаданные платформы, которые хранились в MySQL. Из-за этого пользователи видели устаревшие или противоречивые данные, возникали проблемы с Issues, Pull Requests и комментариями, а также были остановлены webhooks и сборки GitHub Pages.

Ключевой особенностью этого инцидента стало то, что сама исходная проблема длилась всего 43 секунды, но запустила цепочку последствий: автоматическое межрегиональное переключение MySQL, рассинхронизацию между дата-центрами, сложное восстановление из резервных копий и длительное догоняющее выравнивание реплик.

## Влияние на пользователей

Во время инцидента наблюдались следующие проблемы:

- на сайте отображались устаревшие или непоследовательные данные;
- комментарии, изменения в Issues и Pull Requests могли пропадать после обновления страницы;
- операции записи работали медленно из-за роста задержек;
- webhook-события не доставлялись;
- сборки GitHub Pages были поставлены на паузу.

GitHub отдельно сообщил, что данные Git-репозиториев не пострадали. Потери пользовательских данных в обычном смысле не произошло, но часть записей за короткий промежуток времени потребовала дополнительной ручной сверки.

## Причина инцидента

Первичный триггер инцидента произошёл во время плановых работ по замене неисправного 100G-оптического оборудования. В этот момент была потеряна связность между сетевым узлом GitHub на восточном побережье США и основным дата-центром.

Сетевая partition длилась 43 секунды, но за это время Orchestrator, который отвечал за управление топологией MySQL-кластеров, начал процесс автоматического failover. В результате запись была переведена на кластер в западном дата-центре.

Дальше возникла более серьёзная проблема: в восточном дата-центре остались несколько секунд записей, которые не успели попасть на западное побережье до переключения. Поэтому GitHub уже не мог просто вернуть всё назад без риска повредить или потерять часть данных. Именно из-за этого команда выбрала fail forward и дальнейшее восстановление через резервные копии и повторную синхронизацию.

## Хронология инцидента

### 21 октября 2018 года

- **22:52 UTC**  
  Во время сетевых работ произошла потеря связи между сетевым узлом на восточном побережье США и основным дата-центром GitHub. Разрыв длился 43 секунды.

- **22:52 UTC**  
  Orchestrator, работавший в основном дата-центре, начал терять лидерство. Узлы Orchestrator в других площадках собрали quorum и инициировали failover MySQL-кластеров на западное побережье США.

- **После восстановления связи**  
  Прикладной слой GitHub начал направлять записи в новые primary-базы на западном побережье. В этот момент выяснилось, что на восточном побережье осталась небольшая часть записей, которые не успели реплицироваться.

- **22:54 UTC**  
  Внутренний мониторинг начал массово отправлять алерты о неисправностях.

- **23:02 UTC**  
  Дежурные инженеры установили, что топология нескольких MySQL-кластеров находится в нештатном состоянии. По данным Orchestrator было видно, что в активной схеме участвуют только серверы западного дата-центра.

- **23:07 UTC**  
  Команда вручную заблокировала внутренние инструменты деплоя, чтобы исключить дополнительные изменения во время аварии.

- **23:09 UTC**  
  Сайт был переведён в статус `yellow`.

- **23:11 UTC**  
  К инциденту подключился incident coordinator.

- **23:13 UTC**  
  Статус был повышен до `red`. К работе подключили инженеров из database engineering. На этом этапе стало ясно, что проблема затронула несколько кластеров сразу.

- **23:13-23:19 UTC**  
  Команда оценила варианты возврата в исходную схему. Быстрый rollback признали небезопасным, потому что данные между восточным и западным дата-центрами уже разошлись.

- **23:19 UTC**  
  GitHub приостановил доставку webhook-событий и сборки GitHub Pages. Это было сделано сознательно, чтобы не добавлять новые записи в систему, пока топология БД оставалась нестабильной.

### 22 октября 2018 года

- **00:05 UTC**  
  Сформирован план восстановления: поднять кластеры из резервных копий, синхронизировать их с актуальными данными, вернуть стабильную топологию и только после этого разбирать накопившиеся очереди.

- **00:41 UTC**  
  Для всех затронутых MySQL-кластеров запущен процесс восстановления из резервных копий.

- **06:51 UTC**  
  Несколько кластеров в восточном дата-центре уже были восстановлены и начали репликацию данных с западного побережья. Сайт всё ещё работал медленно, но часть чтений уже возвращала актуальные данные.

- **07:46 UTC**  
  GitHub опубликовал расширённое сообщение о характере инцидента и ходе восстановления.

- **11:12 UTC**  
  Все primary-базы снова были размещены на восточном побережье США. Это улучшило отклик сайта, но множество read-replica ещё сильно отставали, поэтому пользователи продолжали видеть неактуальные данные.

- **13:15 UTC**  
  Под дневной нагрузкой стало видно, что задержка репликации не уменьшается, а местами даже растёт. Команда начала поднимать дополнительные read-replica в облаке на восточном побережье, чтобы перераспределить чтения и дать существующим репликам догнать primary.

- **16:24 UTC**  
  После синхронизации реплик GitHub вернулся к стабильной рабочей топологии. При этом статус оставался `red`, потому что оставался большой backlog фоновых задач.

- **16:45 UTC**  
  Началась обработка накопившихся очередей. На этот момент в backlog находилось более 5 миллионов webhook-событий и около 80 тысяч сборок GitHub Pages.

- **Во время обработки backlog**  
  Выяснилось, что около 200 тысяч webhook payload были отброшены из-за истечения внутреннего TTL. После этого обработку временно приостановили и увеличили TTL, чтобы избежать новых потерь в очереди.

- **23:03 UTC**  
  Все pending webhooks и Pages builds были обработаны, корректная работа сервисов подтверждена, статус сайта возвращён в `green`.

## Почему восстановление заняло так долго

Главная причина в том, что GitHub осознанно поставил целостность данных выше скорости восстановления. После переключения между регионами и появления неотреплицированных записей быстрый возврат в исходную схему уже был слишком рискованным.

Кроме этого, длительность инцидента увеличили ещё несколько факторов:

- приложение плохо переносило запись в удалённый регион из-за высокой задержки;
- данные между двумя площадками уже не совпадали;
- восстановление нескольких терабайт из удалённых резервных копий занимало часы;
- read-replica догоняли primary медленнее, чем ожидалось;
- рост дневной нагрузки замедлил выравнивание репликации;
- backlog фоновых задач пришлось обрабатывать отдельно уже после стабилизации базы данных.

## Что было сделано правильно

- Команда быстро остановила деплой и тем самым сократила риск дополнительных изменений во время аварии.
- Решение не делать рискованный rollback помогло сохранить целостность пользовательских данных.
- Восстановление из резервных копий было тяжёлым, но сама процедура была известна и заранее отработана.
- GitHub не закрыл инцидент раньше времени и оставлял статус деградации до тех пор, пока не был разобран backlog.

## Что сработало плохо

- Конфигурация Orchestrator допускала межрегиональное продвижение primary-баз, хотя приложение не было рассчитано на такую схему.
- Система оказалась слишком чувствительной к росту сетевой задержки между приложением и MySQL.
- Оценки времени восстановления были слишком оптимистичными и не учли реальную динамику репликации под нагрузкой.
- Ограничения на TTL во внутренних очередях не были подготовлены к столь длительному инциденту.
- Пользователи долгое время видели неконсистентные данные, что заметно ухудшило восприятие надёжности сервиса.

## Итоги и выводы

Этот инцидент показал, что даже очень короткая сетевая partition в распределённой системе может привести к многочасовому сбою, если автоматическое переключение не согласовано с ограничениями приложений и реальным поведением репликации.

Главный вывод из этого случая в том, что отказоустойчивость нельзя оценивать только по наличию резервных узлов и автоматического failover. Нужно заранее проверять, как система поведёт себя после такого переключения: выдержит ли приложение новую задержку, не разойдутся ли данные между площадками и сколько реально займёт восстановление.

После инцидента GitHub объявил о нескольких изменениях: запретить подобные межрегиональные переключения для Orchestrator, улучшить статус-коммуникацию, ускорить движение к multi-datacenter архитектуре и серьёзнее проверять аварийные сценарии. С инженерной точки зрения это логичные выводы, потому что проблема возникла не из-за одной ошибки, а из-за комбинации архитектурных ограничений и неудачного сценария автоматического failover.
